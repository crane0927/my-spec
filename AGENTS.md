# Repository Guidelines

## 适用范围

本文件定义当前仓库内相对于全局规则新增的默认工作方式，适用于本仓库中的开发、调试、测试、文档整理和轻量自查。未特别说明的部分，继承全局 `AGENTS.md`。

本文件面向默认开发场景，不替代专门的 review skill、security skill 或其他领域 skill。

## 项目定位与目录约定

当前仓库用于实现 `my-spec` 功能，而不只是保存设计文档。现阶段已存在的核心设计文档为：

- `docs/myspec-design.md`：产品与技术设计基线

后续实现时，推荐按职责组织目录：

- `src/`：核心源码
- `tests/`：单元测试、集成测试
- `docs/`：设计文档、实现说明、流程文档
- `scripts/`：开发、校验、发布辅助脚本

新增文件前优先保持结构清晰，避免把实现代码继续堆在仓库根目录。

## 基本原则

1. 优先选择最简单、最清晰、最容易验证的方案，避免提前抽象和不必要的复杂化。
2. 优先复用现有能力，避免重复造轮子和无关重构。
3. 修改尽量聚焦当前目标，采用小步、可回退的外科手术式修改。
4. 保持与现有风格、结构、命名和架构边界一致。
5. 当存在多种实现路径时，优先比较复杂度、风险、可维护性和与现有架构的一致性，再做选择。

## Skill-First 纪律

- 收到任务后，先检查当前已加载 skill 中是否有匹配项。
- 有匹配 skill 时，优先激活并遵循其 workflow、验证方式和输出约定。
- 不要假设某个固定 skill 一定存在，按当前会话实际可用 skill 决策。
- 没有匹配 skill 时，再按本文件的默认流程执行。

## 默认工作流

1. 理解上下文：阅读相关文件、项目结构、需求和验收标准。
2. 实施修改：按最小改动原则编码，过程中做轻量自查。
3. 结果验证：执行最相关的测试、构建、lint 或其他验证步骤。
4. 汇报结果：说明做了什么、如何验证、剩余风险是什么。

## 开发与检查命令

仓库当前尚未提交正式的构建、测试和运行脚本；新增能力时，应同时补齐对应命令，并优先在 `package.json`、`Makefile` 或等价入口中暴露。

当前可用的基础检查命令：

- `ls -la`：查看仓库结构
- `rg --files`：快速列出文件
- `rg -n "^#|^##" docs/*.md *.md`：检查 Markdown 标题层级

开始实现后，应尽快补充并统一以下命令：

- 开发启动命令
- 自动化测试命令
- Lint / 格式检查命令
- 构建命令

## 代码与文档修改要求

- 不要假设技术栈，先查看实际项目文件和依赖声明。
- 不要主动引入新框架、新层级、大规模重构或无关修复，除非任务明确要求。
- 新增或修改代码时，优先保证可读性、一致性和可维护性。
- 关键逻辑应补充符合语言规范的中文注释，说明实现意图、边界处理和注意事项。
- 文档修改应优先补齐使用方式、约束、边界和验收口径，避免空泛描述。

默认使用简体中文编写仓库文档。代码命名与注释可根据技术栈选择中英文，但必须保持一致。

- 目录名使用小写。
- 文件名应表达职责，避免 `temp.*`、`new.*`、`final.*` 之类临时命名。
- 命令、路径、标识符使用反引号包裹。
- 示例命令、配置和结构使用代码块展示。

如果引入具体语言或框架，请同步补充对应的格式化、Lint 和测试约定。

## 审查与路由

- 开发过程中可以做轻量自查。
- 用户明确要求 `review`、代码审查、安全审查或文档审查时，优先使用对应 review / security skill。
- 若当前环境存在专门的 reviewer agent 或等价能力，正式审查应交给该能力处理。

## 验证要求

- 完成修改后，优先执行与改动最相关的验证步骤。
- 如果无法验证，必须明确说明未验证项、原因和风险。

实现 `my-spec` 功能时，还应同步检查：

- 新增或修改的功能是否有对应验证方式。
- 文档中的路径、命令、文件名是否仍然有效。
- 设计文档中的约束是否与实现保持一致。
- 同一功能的代码、测试、文档是否同步更新。

若仓库已引入自动化测试，优先执行与本次改动最相关的测试。

## Git 规则

- 在已确认的计划执行模式下，每完成一个阶段性任务后自动执行一次 `git commit`，无需逐次单独确认。
- 仅在任务边界和验收标准明确时，才可按阶段性任务或 task 粒度自主提交。
- 不要执行高风险、不可逆操作，例如 `rm -rf`、`git reset --hard`、`git push --force`，除非用户明确要求并确认。

提交信息遵循 Conventional Commits，并使用中文，例如：

- `feat(cli): 实现 review 子命令`
- `docs(design): 同步更新验证流程`

Pull Request 或变更说明应至少包含：

- 变更摘要
- 变更原因
- 影响范围
- 已执行的验证步骤

尽量保持一次变更只解决一个明确问题，避免把架构调整、功能开发和文档清理混在同一次提交中。

## 冲突处理

规则优先级如下：

1. 用户当前要求
2. 本仓库内更靠近目标目录的 `AGENTS.md`
3. 本文件
4. 默认通用行为
